# Contributing

Nodus projects ship through reviewed pull requests into `main`.

## Branches

- Use short, descriptive task branches.
- Keep changes scoped to the project or issue being handled.
- Do not commit secrets, local machine paths, tokens, or generated runtime state.

## Pull Requests

- Explain what changed, why it changed, and how it was verified.
- Link related issues or tasks when available.
- Call out risky changes, external service changes, migrations, or follow-up work.

## Verification

Run the relevant project checks before requesting review. If a check cannot be run, state that clearly in the pull request.
